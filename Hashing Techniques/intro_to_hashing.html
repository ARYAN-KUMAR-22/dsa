Current Insight

Hashing uses a formula to jump directly to a bucket. When two keys land in the same bucket, chaining stores them together in a short list.

What Hashing Means

  • Hashing converts a key into an array index using a hash function.
  • That index points to a bucket where the key can be stored.
  • The goal is fast average-time insert, search, and delete operations.

Hash Function In This Demo

  • This intro uses h(k) = k % m, where m is the table size.
  • If k = 44 and m = 7, then 44 % 7 = 2.
  • That sends key 44 to bucket 2.

Why Collisions Happen

  • There are usually fewer buckets than possible keys.
  • Different keys can map to the same bucket index.
  • This page handles collisions with chaining.

Important Terms

  • Bucket: one slot of the table.
  • Load factor: stored keys / table size.
  • Chain length: how many keys live in one bucket.

Collision Handling Options

  • Chaining: keep a small list in the bucket.
  • Open addressing: probe for another empty slot.
  • Chaining is used here because it is easy to visualize.

What Makes A Good Setup

  • Keys should spread out evenly across the table.
  • Too many collisions increase search time.
  • Keeping the load factor under control helps keep chains short.

Intro to Hashing

Hashing turns a search problem into an indexing problem. Instead of scanning all stored items one by one, we compute an index and jump to the bucket where the key should live.

In the average case, that makes insert, search, and delete feel almost immediate, often described as O(1) average time.

Worked Example

Suppose the table size is 7 and the key is 31. We compute 31 % 7 = 3, so the key belongs in bucket 3.

If bucket 3 is empty, the key goes there directly. If it already contains some key like 24, then a collision happens. With chaining, both keys stay in bucket 3, one after the other.

No collision

18 % 7 = 4, bucket 4 is empty, so key 18 becomes the first node there.

Collision

32 % 7 = 4, bucket 4 already has 18, so key 32 is appended to bucket 4's chain.

C Code Example

This sample uses the same idea shown in the animation: index = key % TABLE_SIZE, and collisions are handled by chaining linked nodes inside each bucket.

#include <stdio.h>
#include <stdlib.h>

#define TABLE_SIZE 7

typedef struct Node {
    int key;
    struct Node* next;
} Node;

Node* table[TABLE_SIZE] = {NULL};

int hashFunction(int key) {
    return key % TABLE_SIZE;
}

void insert(int key) {
    int index = hashFunction(key);

    Node* newNode = (Node*)malloc(sizeof(Node));
    newNode->key = key;
    newNode->next = NULL;

    if (table[index] == NULL) {
        table[index] = newNode;
    } else {
        Node* temp = table[index];
        while (temp->next != NULL) {
            temp = temp->next;
        }
        temp->next = newNode;
    }
}

void display() {
    for (int i = 0; i < TABLE_SIZE; i++) {
        printf("Bucket %d: ", i);
        Node* temp = table[i];
        while (temp != NULL) {
            printf("%d -> ", temp->key);
            temp = temp->next;
        }
        printf("NULL\n");
    }
}

int main() {
    int keys[] = {18, 41, 22, 44, 59, 32, 31, 73};
    int n = sizeof(keys) / sizeof(keys[0]);

    for (int i = 0; i < n; i++) {
        insert(keys[i]);
    }

    display();
    return 0;
}

Why Load Factor Matters

The load factor is the ratio of stored keys to table size. A small load factor usually means short chains and fast lookups. A large load factor means more crowded buckets and slower searches.

Real hash tables often resize themselves when the load factor grows too high so the keys spread out again.

Where Hashing Appears In Practice

  • Hash maps: fast key-value storage.
  • Sets: quick membership checks.
  • Caches: store expensive results under a key.
  • Symbol tables: compilers store identifiers efficiently.

Key Takeaways

  • Hashing maps keys to indices for fast average-time access.
  • Collisions are unavoidable, so the table needs a strategy.
  • Chaining is simple to understand and easy to visualize.
  • The hash function and load factor strongly affect performance.
